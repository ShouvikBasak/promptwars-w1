# Cloud Firestore Data Model

This document outlines the minimal Cloud Firestore data model for the StadiumFlow assistant. It adheres strictly to the defined architecture constraints: flat collections, no complex relationships, heavy reliance on server-side aggregation, and strict data lifecycles.

## 1. Collection: `pois` (Points of Interest)

**Purpose:** 
Stores the defining metadata for venue locations (restrooms, concessions, exits) along with their live, aggregated wait-time state. As defined in `ARCHITECTURE.md`, attendee UI clients will subscribe directly to these aggregated documents rather than polling raw inputs.

**Document Fields:**
- `id` *(String)*: Unique identifier for the POI.
- `name` *(String)*: Human-readable display name (e.g., "Section 108 Restroom").
- `type` *(String)*: Category of the POI (valid values: `restroom`, `concession`, `exit`).
- `zone` *(String)*: The manual location grouping (e.g., "North Concourse") used to filter views, aligning with the explicit location selection requirement.
- `currentWaitMinutes` *(Number)*: The active wait time, generated by Cloud Run aggregation or direct staff override.
- `confidenceScore` *(Number)*: The calculated time-decay reliability score (0.0 to 1.0) based on recent inputs.
- `isStaffOverride` *(Boolean)*: Flag indicating if the `currentWaitMinutes` was set manually via the `/ops` console, enforcing the rule that staff validation overrides aggregation.
- `lastUpdatedAt` *(Timestamp)*: The exact time the aggregated state was last computed.

---

## 2. Collection: `wait_signals`

**Purpose:** 
Stores the raw, ephemeral wait-time inputs submitted by event attendees or staff. This collection acts as the input layer for the Cloud Run aggregation logic and is intentionally isolated from client reads.

**Document Fields:**
- `id` *(String)*: Unique identifier for the signal.
- `poiId` *(String)*: The ID of the point of interest this signal applies to.
- `waitMinutes` *(Number)*: The estimated wait time specified by the user.
- `submitterRole` *(String)*: Determines the source of the input (`attendee` or `staff`), allowing server logic to route staff-submitted signals directly into overrides.
- `createdAt` *(Timestamp)*: The time the signal was recorded.
- `expiresAt` *(Timestamp)*: The calculated Time-To-Live (TTL) expiration time (strictly `createdAt + 15 minutes`). Used natively by Firestore TTL policies to purge stale data and ensure AI inputs are always fresh.

---

## 3. Collection: `broadcasts`

**Purpose:** 
Supports the "minimal ops console" functionality by allowing venue staff to push short, localized announcements to specific venue zones.

**Document Fields:**
- `id` *(String)*: Unique identifier for the message.
- `zone` *(String)*: The venue area this message targets (e.g., "South Gate"), mapping to the attendees' manual location context.
- `message` *(String)*: The plain-language text content of the broadcast.
- `createdAt` *(Timestamp)*: The exact time the message was published by staff.
- `expiresAt` *(Timestamp)*: An expiration time specifying when the message should be removed from the attendee UI, preventing old alerts from misguiding attendees.

---

### Design Validation
- **Fewest Collections Possible:** Maintained at 3 minimal collections.
- **Flat Structure:** Absolutely no subcollections; all documents sit neatly at the root level.
- **No Optional Fields:** Every property across all three collections is strictly required to process aggregation, display localized context, or trigger TTLs.
- **No Code Generation:** Defined logically and strictly through typed definitions.
